## ⚠️ Важно про хостинг
Обычный статический хостинг (просто «закинуть файлы») подойдёт **только для `public/`**, но парсер и push-рассылка — это `server.js`, который должен работать **постоянно** (как процесс Node.js), а не просто отдавать файлы. Статический хостинг (например, обычный shared-хостинг или GitHub Pages) для этого не годится.

Варианты:
1. **Всё вместе на одном Node-хостинге** (проще всего): Render.com, Railway.app, Fly.io — все имеют бесплатный/дешёвый тариф с поддержкой постоянно работающих Node-процессов. Сервер сам отдаёт и `public/`, и API — грузите весь проект целиком.
2. **VPS** (timeweb, reg.ru, Selectel и т.п.) — запустить `node server.js` через pm2/systemd, чтобы процесс не останавливался.
3. Если у вас уже есть только статический хостинг для фронтенда — задеплойте `server.js` отдельно (вариант 1 или 2), а в `public/index.html` впишите его адрес в константу `API_BASE`, например:
   ```js
   const API_BASE = 'https://ваш-backend.onrender.com';
   ```

## Установка и запуск локально
```bash
npm install
npm start
```
Откройте `http://localhost:3000`.

## Настройка push-уведомлений (VAPID)
Сгенерируйте постоянную пару ключей (иначе при каждом перезапуске сервера все подписки слетят):
```bash
npx web-push generate-vapid-keys
```
Задайте их как переменные окружения на хостинге:
```
VAPID_PUBLIC_KEY=...
VAPID_PRIVATE_KEY=...
```
Если их не задать, сервер сгенерирует временные ключи сам и выведет их в консоль при старте — это нормально для теста, но не для продакшена.

## Push на iPhone — важные ограничения Apple
- Работает **только в Safari**, начиная с **iOS 16.4**.
- Сайт должен быть **добавлен на экран «Домой»** («Поделиться» → «На экран «Домой»») и **открыт хотя бы один раз** как отдельное приложение — из обычной вкладки Safari подписка на push недоступна.
- После этого разрешение на уведомления запрашивается так же, как в браузере — через кнопку «Включить уведомления» в приложении.

## Push на Android
Работает в Chrome/Edge/Firefox как в обычной вкладке, так и после установки на домашний экран — без дополнительных ограничений.

## Как работает парсер
- Каждые 10 секунд сервер запрашивает `https://t.me/s/mchs31` (открытая веб-версия канала, авторизация не нужна).
- Сообщения классифицируются по ключевым словам: ракетная опасность / БПЛА / отбой / укрытие / повторно.
- Регион определяется по упоминанию города/района в тексте; если не найден — по умолчанию «Белгород».
- Сообщения, похожие на рекламу (по ключевым словам и признакам), отбрасываются.
- Новые сообщения сразу уходят подписчикам через Web Push с учётом выбранных ими районов.

## Ограничения парсера, о которых стоит знать
- Классификация по ключевым словам — эвристика. Если МЧС31 изменит формулировки, часть сообщений может попасть в категорию «Сообщение канала» вместо конкретного типа — стоит периодически сверяться с реальной лентой и донастраивать `REGION_KEYWORDS` / классификатор в `server.js`.
- `t.me/s/<channel>` — неофициальный, но общедоступный способ чтения каналов; если Telegram изменит вёрстку этой страницы, потребуется поправить селекторы в `fetchChannelMessages()`.
